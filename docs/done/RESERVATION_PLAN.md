# 예약 시스템 구현 방안

> 단골 반복 예약 (템플릿) + 일회성 예약 + 예약 → 주문 전환 흐름.
> 관련 테이블: `m_order_rsv_tmpl`, `m_order_rsv_menu`, `t_order_rsv`, `t_order_rsv_menu`, `t_order.rsv_seq`.

## 1. 핵심 원칙 (결정된 사항)

1. **템플릿 → 인스턴스 자동 생성은 WebSocket 으로 push** — 백엔드 cron 이 `t_order_rsv` 를 생성하면 클라이언트에 push. 프론트는 polling 하지 않음
2. **예약 접수 = `t_order` 생성** — 예약을 "접수" 하면 반드시 주문 엔티티가 생성됨. 이후 `OrderStatusPage` 에 "예약에서 생성됨" 으로 표시
3. **예약 페이지는 주문현황과 동일한 2단 구조**:
   - 상단: 진행 중 예약 (RESERVED 상태)
   - 하단: 처리 이력 (접수 COMPLETED / 취소 CANCELED) — **되돌리기 가능 → 다시 RESERVED 로**
4. **당일 예약 중심** — 템플릿이 자동 생성하므로 대부분 "오늘 것" 만 보임. 내일/이번주 예약은 별도 탭/필터
5. **예약/템플릿 생성 시 매장·메뉴 선택은 `OrderPage` 와 동일 UI/프로세스** — 좌측 매장 탐색 → 중앙 메뉴 그리드 → 우측 추가 필드 패널 (차이점은 우측 패널 구성)

## 2. 페이지 구성

| 그룹 | 파일                              | 라우트                                      | 설명                                                       |
| ---- | --------------------------------- | ------------------------------------------- | ---------------------------------------------------------- |
| 영업 | `ReservationsPage.vue`            | `/order-rsvs`                               | 당일 진행 중 예약 + 처리 이력 (주문현황과 유사)            |
| 영업 | `ReservationCreatePage.vue`       | `/order-rsvs/new`                           | 일회성 예약 생성 (주문 페이지 동일 + 우측 예약필드 패널)   |
| 관리 | `ReservationTemplatesPage.vue`    | `/reservation-templates`                    | 템플릿 목록/필터                                           |
| 관리 | `ReservationTemplateEditPage.vue` | `/reservation-templates/new` · `/:seq/edit` | 템플릿 추가/수정 (주문 페이지 동일 + 우측 템플릿필드 패널) |

## 3. 영업 > 예약 관리 (`ReservationsPage`)

### 구조 (주문현황과 미러링)

```
┌──────────────────────────────────────────────────────────┐
│ 예약 관리              [오늘 ▾] [매장 ▾] [+ 예약 추가]    │
├──────────────────────────────────────────────────────────┤
│ ● 진행 중 예약 (5)                                        │
│ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐              │
│ │샘표 11:30│ │세림 12:00│ │...      │ │...      │           │
│ │🕐임박    │ │메뉴 ..   │ │         │ │         │           │
│ │[접수][X] │ │[접수][X] │ │         │ │         │           │
│ └────────┘ └────────┘ └────────┘ └────────┘              │
│                                                          │
│ ● 처리 이력 (접수 · 취소) — 당일                         │
│ ┌──────────────────┐ ┌──────────────────┐               │
│ │✓ 샘표 09:30 접수 │ │✕ 세림 10:00 취소  │               │
│ │ [진행으로 복구]  │ │ [진행으로 복구]   │               │
│ └──────────────────┘ └──────────────────┘               │
└──────────────────────────────────────────────────────────┘
```

### 카드 스타일

- **진행 중 카드**: `OrderStatusPage` 주문 카드와 동일한 외형 (border 색상 + 임박도 chip). 차이점만:
  - 제목: 매장명 + 예약시각 (크게 표시)
  - 임박도: 예약시각까지 남은 분. 15분 이내 `caution`, 5분 이내 `danger`
  - chip: `[#템플릿-샘표정식]` or `[#일회성]` — 출처 표시
  - 액션: `[접수]` (primary) · `[취소]` (secondary danger)
- **처리 이력 카드**: 회색 처리 + 처리시각 + 복구 버튼
  - `[진행으로 복구]` — CANCELED/COMPLETED 상태를 다시 RESERVED 로
  - 주문현황의 "완료 취소 (30초 내)" 와 유사하지만, 예약은 **시간 제한 없이 되돌리기 가능** (기본값, 아래 3-1 결정 필요)
  - 단 `COMPLETED` 복구 시 이미 생성된 `t_order` 는 어떻게 할지 결정 필요 (3-2)

### 상단 필터

- 날짜 (기본: 오늘 / 내일 / 이번주)
- 매장
- 상태 (모두 / 진행 중 / 처리 완료)

### WebSocket 연동

- `/api/reservation` 채널 listen:
  - `POST` (새 인스턴스 생성) → 진행 중 섹션에 추가
  - `PATCH` (status 변경) → 진행 중 ↔ 이력 이동
  - `DELETE` → 양쪽에서 제거
- 주문현황과 같은 패턴 재사용

## 4. 예약 접수 흐름 (가장 중요)

```
[진행 중 예약] 카드의 [접수] 클릭
    ↓
1. PATCH /api/reservation/:seq  { status: COMPLETED }
    ↓
2. 백엔드: INSERT INTO t_order (..., rsv_seq = :seq) + INSERT INTO t_order_menu
    ↓
3. WS push:
   - /api/reservation PATCH  (예약 상태 업데이트)
   - /api/order POST          (새 주문 생성)
    ↓
4. 프론트:
   - ReservationsPage: 진행 중 → 이력 이동
   - OrderStatusPage (열려 있으면): 진행 중 주문에 새 주문 추가
     · 카드에 "📅 예약에서 생성" chip 표시
```

**취소 흐름**: `PATCH /api/reservation/:seq { status: CANCELED }` — 주문 생성 안 함. 이력에만 표시

**복구 흐름**: `PATCH { status: RESERVED }` — COMPLETED 였다면 연결된 `t_order` 처리 필요 (3-2 결정)

## 5. 예약 생성 — 일회성 (`ReservationCreatePage`)

**주문 페이지와 동일한 3-컬럼 레이아웃 + 우측 패널만 차이**

```
┌─ 매장 선택 ─┬─ 메뉴 선택 ────────────────┬─ 예약 정보 + 요약 ────┐
│            │                            │                      │
│ [탭들]     │ [메뉴 그리드]              │ 매장: 샘표            │
│ - 중앙     │ 돈까스 · 제육 · 갈비탕 …   │                      │
│ - 농협     │                            │ 📅 예약 일시 *        │
│ - 원예     │                            │   2026.04.23 11:30   │
│ ...        │                            │                      │
│            │                            │ 📝 비고              │
│ 매장 카드  │ 선택된 메뉴 수량 토글       │   (선택)             │
│ ...        │                            │                      │
│            │                            │ ─────                │
│            │                            │ 담긴 메뉴 (2)        │
│            │                            │ · 돈까스 ×2 16,000   │
│            │                            │ · 제육 ×1   8,000   │
│            │                            │ ─────                │
│            │                            │ 총 금액: 24,000원    │
│            │                            │                      │
│            │                            │ [취소] [예약 등록]    │
└────────────┴────────────────────────────┴──────────────────────┘
```

**우측 패널의 "예약 정보" 섹션만 추가** — 주문 페이지의 우측 패널 상단에 예약 일시/비고 필드가 들어가고, 나머지(담긴 메뉴, 합계, CTA)는 동일

## 8. 주문현황 연계

- `OrderStatusPage` 주문 카드에서 **출처 chip** 표시:
  - 일반 주문: chip 없음
  - 예약 유래 (`t_order.rsv_seq IS NOT NULL`): `📅 예약` chip (파란색)
- 카드 클릭/hover 시 원 예약 정보 툴팁 (시각, 템플릿명)

## 9. WebSocket 채널 정리

| 채널               | 이벤트            | 구독 페이지                                                  |
| ------------------ | ----------------- | ------------------------------------------------------------ |
| `/api/reservation` | POST/PATCH/DELETE | ReservationsPage                                             |
| `/api/order`       | POST/PATCH/DELETE | OrderStatusPage · ReservationsPage (예약 → 주문 전환 감지용) |

ReservationsPage 는 두 채널 모두 구독. 예약 접수 시 서버가 먼저 `reservation PATCH` push, 이어서 `order POST` push → 클라이언트는 두 이벤트 순서대로 처리.

## 10. 추가 결정 필요 항목

| #     | 항목                                                | 선택지                                                                                                                     |
| ----- | --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| 10-1  | 처리 이력 "복구" 시간 제한                          | (a) 무제한 (b) 당일만 (c) N분 이내 (주문현황은 30초)                                                                       |
| 10-2  | COMPLETED → RESERVED 복구 시 연결된 `t_order` 처리  | (a) 같이 삭제 (b) t_order 는 유지하고 예약만 되돌림 (데이터 정합성 주의) (c) COMPLETED 는 복구 불가, CANCELED 만 복구 가능 |
| 10-3  | 예약 시각 임박 알림                                 | (a) 카드 색상 변화만 (b) 토스트 (c) 사이드바 배지 (d) 여러 개 조합                                                         |
| 10-4  | 예약 삭제 vs 취소 구분                              | 상태 `CANCELED` 만 쓰고 물리 삭제 제공 X (추천) / 물리 삭제도 허용                                                         |
| 10-5  | 템플릿 비활성 (`active=false`) vs 종료일 (`end_dt`) | (a) 둘 다 사용 — active 는 즉시 중단, end_dt 는 예정된 종료 (b) 하나로 통일                                                |
| 10-6  | 템플릿 편집이 기존 인스턴스에 미치는 영향           | (a) 미래분만 반영 (추천, 과거는 snapshot) (b) 미처리 인스턴스 전체 갱신                                                    |
| 10-7  | 일회성 예약 인라인 추가 위치                        | (a) 별도 페이지 `/order-rsvs/new` (b) `ReservationsPage` 내 Dialog (c) 두 방식 모두                                        |
| 10-8  | 내일/이번주 예약 노출 방식                          | (a) 필터만 (b) 탭 분리 "오늘/내일/이번주" (c) 토글                                                                         |
| 10-9  | "📅 예약" chip 스타일                               | chip 색상, 위치, 호버 툴팁 내용                                                                                            |
| 10-10 | 템플릿 삭제 시 생성된 인스턴스                      | (a) 연결만 끊음 (rsv_tmpl_seq=NULL) (b) 함께 삭제 (c) 활성 템플릿만 삭제 차단                                              |

## 11. 구현 우선순위 제안

1. `ReservationsPage` 조회 + WS 구독 (읽기 전용)
2. 예약 접수 → t_order 생성 흐름 (`PATCH /api/reservation` + t_order WS 처리)
3. 예약 취소 + 복구
4. `ReservationCreatePage` (일회성)
5. `OrderStatusPage` 에 "예약" chip 표시
6. `ReservationTemplatesPage` 목록
7. `ReservationTemplateEditPage` 편집
8. 내일/이번주 필터 확장
